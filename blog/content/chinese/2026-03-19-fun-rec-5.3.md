# Fun Rec 5.3  LLM建模的基本流程 读书笔记

原文链接 [https://datawhalechina.github.io/fun-rec/chapter_5_gr_basic/3.llm_base.html](https://datawhalechina.github.io/fun-rec/chapter_5_gr_basic/3.llm_base.html)

大语言模型（LLM）的核心原理是，**经过大规模预训练后，模型展现出了强大的语言能力**。

# 第一步：预训练

### 基本方式
是使用**自监督学习**，让模型在大量文本数据中学习通用的语言表达能力。自监督学习的核心，就是根据上文预测下一个词，让模型掌握语言的基本规律和世界知识。

### 预训练数据
- 预训练的数据来源非常广泛，包括网页、书籍、学术论文、代码和各类对话等。

预训练数据有两点核心要求：
- **多样性**
- **高质量**

预训练的数据量级通常从数十亿到数万亿的token不等。

# 第二步：指令微调
预训练之后是**指令微调**阶段，目的是让模型学会**理解并执行具体任务**，使其成为一个真正具备完成任务能力的模型。

### 基本方法
指令微调的数据通常需要包含**指令**和**输入**，在执行过程中，损失函数只对模型的**输出**部分进行计算，指令和输入部分的损失不参与梯度和模型参数的更新。这样模型会更专注于学习如何根据给定的指令和输入来生成正确的回答。

以新闻推荐、网红内容推荐等场景为例，对于一个指令，可能存在一条或多条预期的优质结果。经过指令微调后，模型在这些任务上的表现会显著提高。

# 第三步： 偏好对齐

- 指令微调虽然让模型学会了遵循指令，但其生成的內容仍然可能存在一些问题，比如说**有用性不足**、**事实性错误**等。
- 偏好对齐的目标是让模型的输出更好地符合人类价值观和偏好，而不仅仅是模仿“人类会怎样回答这个问题”。其核心区别在于，监督微调的定位是对人类回答的模仿，效果可能还不够好。因此，基于人类反馈的交互式学习，成为了进一步提升模型能力的关键方法。

### 基本原理
 **基于人类反馈的强化学习** （Reinforcement Learning from Human Feedback, RLHF）。

RLHF的第一步是**收集人类偏好数据**。整体分为三个核心步骤：
1.  **收集人类对模型输出的偏好排序**，形成成对的比较数据（例如，明确指出哪个输出更好，哪个更差）。
2.  **训练一个奖励模型**（Reward Model），让它学习并量化人类的偏好标准。
3.  **使用强化学习**（如PPO算法）策略进行优化，让模型的输出目标变为最大化奖励模型的评分。
RLHF的完整技术路径是：**人类反馈 → 奖励模型 → 策略优化**。

另一种更简洁高效的方法叫 **DPO（直接偏好优化）** ，其核心思想是**直接**根据偏好数据优化策略模型，而无需显式地训练一个独立的奖励模型。它在多项任务中取得了与RLHF相当甚至更优的效果。

> DPO 似乎取代RLHF 成为新标准？

经过偏好对齐后的大语言模型，其输出通常会有以下改进：
-   **回答更简洁、更切题。**
-   **事实准确性更高。**
-   **幻觉（即编造信息）更少。**
-   **更好地遵循复杂指令和用户的隐含意图。**

### 平衡语言模型能力与协同过滤信号

在将LLM应用于推荐系统时，一个核心挑战是平衡其自身强大的语言能力与推荐系统关键的协同过滤信号。
- 如果模型的语言能力过强，可能会过度依赖物品的内容特征，而忽略了从用户行为中习得的协同过滤信号；
- 反之，如果过于聚焦协同信号，又可能无法充分发挥语言模型的语义理解优势。

几点注意事项

- 一个关键思路是将各个推荐任务统一至“理解指令-生成回复”的范式中，即将**推荐任务指令化**。这一过程的挑战在于，推荐系统中的物品往往以**ID**形式存在，而纯粹的ID对语言模型而言不包含任何语义信息，这为指令化带来了困难。
- 此外，在推荐系统的反馈体系中，用户的反馈通常是**隐式**的（如点击、观看时长），没有明确的好坏标签；
- 同时，业务目标往往是多维度的，需要在多个指标间取得平衡。
  **如何从隐式反馈中构建有效的训练信号，以及如何在多目标优化中找到平衡点**，这两点是在模型对齐过程中非常重要的环节。

### 推荐场景中的特殊挑战：

1.  **物品Token化**：物品如果只用ID表示，对语言模型而言毫无意义。如何将ID的语义信息注入模型，使其能够理解ID背后的含义，是一个需要特化的关键过程。
2.  **信号的融合**：推荐系统的强大之处在于协同过滤。如何在平衡语言模型能力的基础上，有效融入协同过滤信号，使得大语言模型能够出色地完成推荐任务，这需要我们在模型设计和训练策略上进行非常精细的考量。
3.  **冷启动问题**：语言模型强大的语义理解能力，可以为新物品的文本描述等内容信息快速建立推荐能力，这是其巨大优势。如何设计模型，使其在有交互数据时主要依赖协同数据，在无交互数据时能灵活切换为依赖内容理解，这将大大缓解推荐系统的冷启动问题。
4.  **实时性要求**：在线推荐系统要求在毫秒级完成推理，而大语言模型自身的自回归生成机制导致其推理延迟可能较高。如何通过系统架构优化、模型加速等方式来显著降低延迟，是决定其能否在工业界落地的关键问题。

最后一点总结：**将大语言模型应用于推荐，不是简单地将语言模型套用到推荐任务上，而是要将推荐问题重新概念化为一个序列生成问题，并根据推荐场景的独特性进行深度匹配和改造。**
