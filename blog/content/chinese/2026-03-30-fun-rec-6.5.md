## 6.5 OneTrans: 统一序列与特征交叉 读书笔记

[原文链接](https://datawhalechina.github.io/fun-rec/chapter_6_scaling/5.one_trans.html)

RankMixer 通过 hardware-aware 架构设计解决了 GPU 利用率问题，但推荐系统的整体架构仍存在碎片化。工业界主流的推荐模型普遍采用 **encode-then-interaction** 范式：序列建模模块将行为序列编码为固定长度向量，然后与非序列特征拼接，输入特征交互模块学习高阶交叉。

这种分离式设计存在两个根本问题：
1. **信息流受限**：序列必须压缩为固定维度向量，静态特征无法在序列编码阶段发挥作用
2. **执行碎片化**：两模块独立执行，无法享受 LLM 的系统优化（KV Caching、FlashAttention）

OneTrans 提出根本性架构革新：**用单一 Transformer backbone 同时完成序列建模和特征交互**。

### 6.5.1 统一 Tokenization

推荐系统的输入包含两类截然不同的特征：**序列特征** \(\mathcal{S}\)（用户的多行为序列）和**非序列特征** \(\mathcal{NS}\)（静态属性和上下文）。

OneTrans 的核心创新是：**将两类特征统一转换为 token 序列，在同一个 Transformer 中处理**。对于序列特征，用行为特定的 MLP 将每个序列对齐到统一维度后合并。支持两种融合策略：
1. **Timestamp-aware**：按时间顺序交错排列所有行为，并添加行为类型标识符
2. **Timestamp-agnostic**：按行为意图强度排序，并在不同行为序列之间插入可学习的 `[SEP]` token

对于非序列特征，OneTrans 将所有特征 concat 后通过单个 MLP 投影，然后切分为多个 token（**Auto-Split Tokenizer**）。

### 6.5.2 Mixed Parameterization

如果直接用标准 Transformer 处理统一 token 序列，会遇到**token 的异质性**问题。S-tokens 来自行为序列（同质性强），而 NS-tokens 来自完全不同的特征空间。

OneTrans 的核心创新是 **Mixed Parameterization**：**S-tokens 共享一套参数，每个 NS-token 拥有独立的 token-specific 参数**。这种设计基于两个观察：行为序列中的所有事件都在同一语义空间，可以高效地共享参数学习序列模式；非序列特征来自异构空间，需要独立参数捕捉各自的特性。

OneTrans 采用 **Causal Attention Mask**，NS-tokens 放置在 S-tokens 之后，导致三个关键的信息流模式：
1. **S-side 因果依赖**：每个 S-token 只能 attend 到它之前的 S-tokens，自然建模时间因果关系
2. **NS-side 全局 attention**：每个 NS-token 可以 attend 到所有 S-tokens（完整行为历史）
3. **支持 Pyramid**：Causal mask 的方向性使得信息自然向序列尾部聚集

### 6.5.3 Pyramid Stack 渐进式蒸馏

OneTrans 的 Causal Attention 有一个重要特性：**信息自然向序列后方聚集**。随着层数加深，靠后的 token 逐渐成为前面所有 token 信息的"汇聚点"。

Pyramid Stack 正是利用这一特性：**逐层减少参与 attention 计算的 query token 数量，只保留序列尾部的 token**。这种设计带来两个核心收益：
1. **Progressive Distillation**：长行为序列通过逐层收缩，将信息逐步"蒸馏"到少量尾部 token 中。浅层学习局部模式，深层学习全局模式
2. **Compute Efficiency**：将 attention 复杂度从 \(O(L^2 d)\) 降低到 \(O(LL'd)\)

### 6.5.4 Cross-Request KV Caching

统一架构的一个关键优势是可以无缝应用 LLM 的系统优化，其中最重要的是 **KV Caching**。在推荐场景中，一次请求通常返回数百个候选物品，这些样本的**用户侧特征完全相同**，只有物品侧特征不同。

OneTrans 的统一 Transformer 自然地支持两阶段计算：
1. **Stage I（S-side）**：处理所有 S-tokens，计算每一层的 key、value 并缓存，每个请求只执行一次
2. **Stage II（NS-side）**：对于每个候选物品，计算其对应的 NS-tokens，使用缓存的 S-side keys 和 values

传统方法需要 \(O(C \cdot L_S)\) 的序列计算，而 KV Caching 将其降低到 \(O(L_S + C \cdot L_{NS})\)。由于 \(L_{NS} \ll L_S\)，实际复杂度近似 \(O(1)\)（相对于候选数 \(C\)）。

更进一步，OneTrans 实现了 **Cross-Request KV Caching**。用户的行为序列是 append-only 的，每次新请求相比上一次请求，只是在序列末尾新增了少量行为事件，因此可以在多个请求之间复用 KV cache。

### 6.5.5 统一建模的本质

OneTrans 的核心贡献在于推荐系统架构的根本性转变：**从模块组合到统一建模**。传统 encode-then-interaction 将序列编码和特征交互分离为独立模块，不同类型的交互被人为隔断。OneTrans 通过统一 Transformer，让这些交互在每一层同时发生。

统一架构带来的另一个关键优势是**整体可扩展性**。分离架构需要分别调优序列模块和交互模块，难以形成统一的 scaling law。OneTrans 将整个模型统一为单一 Transformer backbone，扩展策略变得简单明确：增加层数（depth）、增加隐藏维度（width）、增加序列长度（length）。

从 RankMixer 到 OneTrans，推荐系统架构演进的两个关键方向清晰可见：hardware-aware 的计算设计解决了 GPU 利用率问题，统一建模框架打破了模块碎片化壁垒。两者的结合为推荐系统走向大规模、可扩展的智能化奠定了基础。
