## 6.3 MTGR：混合范式建模 读书笔记

[原文链接](https://datawhalechina.github.io/fun-rec/chapter_6_scaling/3.mtgr.html)

HSTU证明了推荐模型可以遵循 scaling law，GenRank揭示了生成式推荐的本质是自回归机制而非训练范式。但两者的纯粹性是有代价的：为了实现完整的行为序列建模，模型在预测用户对候选物品的行为时，不能使用任何候选相关的**交叉特征（cross features）**。美团团队实验发现，去除交叉特征会导致模型性能显著下降，即使通过大幅增加模型规模也无法弥补。

MTGR（Meituan Generative Recommendation）提出了一种**混合范式**：在保持用户粒度聚合效率的同时，支持 target-aware 的判别式建模。

### 6.3.1 范式的再思考：生成 vs 判别

HSTU 和 GenRank 都采用交织式建模，将推荐过程建模为内容和行为交替的序列：

\[[\Phi_0, a_0, \Phi_1, a_1, \ldots, \Phi_{n_c-1}, a_{n_c-1}]\]

排序任务对应于建模 \(p(a_i | \Phi_0, a_0, \ldots, \Phi_i)\)。但问题在于，候选 \(\Phi_i\) 是序列的一部分，与历史行为 \(a_{i-1}\) 地位平等。用自回归方式训练时，位置 \(i\) 的预测只能依赖位置 \(0\) 到 \(i-1\) 的信息，而**交叉特征**往往需要"跨越"这个顺序依赖——它们需要同时看到用户的某个历史统计和当前候选的属性，然后计算两者的交互。

在纯生成式建模下，这种跨越是被禁止的。如果允许候选 \(\Phi_i\) 的表示依赖于"用户对该候选所属类目的历史偏好"这样的特征，就破坏了序列建模的因果性，因为这个特征实际上已经"看到了未来"。

> 注释：交叉特征包括"用户对该类目的历史点击率"、"用户在该时段对此类内容的偏好"等，它们精确捕捉了用户与候选之间的细粒度交互模式。

#### 6.3.1.2 判别式排序的本质

为什么交叉特征如此关键？排序阶段是一个典型的**判别式任务**：给定输入 \(x\)（历史+候选），预测标签 \(y\)（行为）。传统 DLRM 将其形式化为 \(p(a | u, i)\)，关键在于用户表示 \(u\) **可以依赖于候选物品** \(i\)。比如"用户对科技类内容的平均点击率"这个特征，只有在候选是科技类物品时才有意义，这是 \(u \times i\) 的二阶交叉。

判别式方法关心的是 \(p(a | 历史, 候选)\) 的条件分布，不需要建模完整的联合分布。生成式方法通过分解联合分布来得到条件分布，但带来了额外负担——你必须建模 \(p(候选 | 历史)\)，即使这不是你真正关心的。

#### 6.3.1.3 MTGR 的核心洞察

MTGR 提出了一个看似矛盾实则巧妙的方案：使用生成式模型的架构（Transformer + 用户粒度聚合），但保持判别式的建模目标。将同一用户的多个候选聚合到一个样本中：

\[[\text{User}, \text{Seq}, \text{RealTime}, [\text{Cross}_1, \text{Item}_1], [\text{Cross}_2, \text{Item}_2], \ldots]\]

关键差异在于：历史部分（User, Seq, RealTime）与 HSTU/GenRank 一致，是用户的完整行为序列；而候选部分（Cross, Item）不再是历史的延续，而是待预测的目标，且每个候选的表示中**直接包含了交叉特征**。

从 token 组织的角度：
- **User tokens**：用户的静态属性（年龄、性别、城市等）
- **Sequence tokens**：用户的历史交互序列（长期行为）
- **RealTime tokens**：用户的近期交互（短期行为）
- **Candidate tokens**：每个候选一个 token，融合了物品特征和交叉特征

这种组织仍保留了用户粒度聚合的核心优势：对于 \(m\) 个候选，用户历史部分只需编码一次，\(m\) 个候选 token 并行处理。计算复杂度是 \(O((n + m)^2)\) 而非 \(O(m \cdot n^2)\)，当 \(m \ll n\) 时带来显著效率提升。

这个设计的哲学是：**区分手段和目的**。生成式架构是强大的表征学习手段，但不一定要服务于生成式的建模目标。用户粒度聚合是高效的计算组织方式，但不一定要求完整的因果序列。

### 6.3.2 架构创新的原理

混合范式的思想清晰，但实现面临挑战：当历史序列和候选物品放在同一个 Transformer 中处理，但承担不同角色时，如何确保模型有效学习？

#### 6.3.2.1 特征到 Token 的映射

MTGR 为每个候选构造一个独立的 token，融合了物品固有特征、交叉特征和位置时序信息：

\[\text{CandidateToken}_i = \text{MLP}(\text{Concat}(\text{Emb}(\text{Item}_i), \text{Emb}(\text{Cross}_i)))\]

关键设计决策：**交叉特征被视为候选表示的一部分，而不是历史序列的一部分**。用户历史部分的 token 则是"纯粹的"，不依赖任何候选信息，只编码历史。

但这种非对称 token 组织带来了问题：不同类型的 token 在语义上处于不同的空间。用户属性 token 编码人口统计学信息，序列 token 编码行为模式，候选 token 编码物品特征加交叉信息，直接用统一 Transformer 处理会相互干扰。

#### 6.3.2.2 Group Layer Normalization

标准 LayerNorm 在每个 token 的特征维度上归一化：

\[\text{LayerNorm}(x) = \frac{x - \mu}{\sigma} \cdot \gamma + \beta\]

它假设所有 token 共享相同的特征分布。但在 MTGR 中，这个假设被打破了——User token、Sequence token、Candidate token 的激活值分布有显著差异，甚至相同维度编码的语义完全不同。

MTGR 提出 **Group Layer Normalization (GLN)**，将 token 分为四组独立归一化：

- Group 1: User tokens
- Group 2: Sequence tokens
- Group 3: RealTime tokens
- Group 4: Candidate tokens

\[\text{GLN}(x_i) = \frac{x_i - \mu_{g(i)}}{\sigma_{g(i)}} \cdot \gamma_{g(i)} + \beta_{g(i)}\]

其中 \(g(i)\) 表示 token \(i\) 所属的 group。这样做有两个好处：**第一，分布对齐**——每个 group 内部的 token 语义相近，独立归一化更好地稳定训练；**第二，语义独立**——不同 group 的 token 在相同维度上可以编码不同信息，归一化参数的独立性保证了语义的独立性。

#### 6.3.2.3 Dynamic Masking

HSTU 和 GenRank 使用 causal mask（下三角 mask），确保位置 \(i\) 只能看到之前的信息。但在 MTGR 中，token 序列不再严格按时间顺序组织：

\[[\text{User}, \text{Seq}, \text{RealTime}, \text{Cand}_1, \text{Cand}_2, \ldots, \text{Cand}_m]\]

如果简单使用 causal mask，候选之间会产生不合理的可见性（Cand\(_2\) 能看到 Cand\(_1\)）。更复杂的是 RealTime tokens，它们的时间戳可能与候选曝光时间重叠，简单聚合会导致**信息泄露**。

MTGR 提出 **Dynamic Masking**，通过三条规则实现细粒度的可见性控制：

1. **静态序列对所有 token 可见**：User 和 Seq tokens 来自聚合窗口之前的历史，任何候选都可以 attend 到这些信息。
2. **动态序列遵循因果性**：RealTime tokens 之间的可见性取决于时间戳，对 Candidate tokens 的可见性根据实际时间戳动态决定。
3. **候选之间相互独立**：Cand\(_i\) 对 Cand\(_j\)（\(j \neq i\)）不可见，保证评分的独立性。

这个 mask 不是预先固定的，而是根据每个样本中 token 的实际时间戳**动态生成**的。Dynamic Masking 体现了 MTGR 对时序建模的细致理解：对真正静态的信息允许全局可见，对动态信息严格遵守因果性，对并行目标保持独立性。

Dynamic Masking 是 MTGR 混合范式的最后一块拼图。它让模型在统一的 Transformer 架构中，同时处理因果序列（历史行为）和非因果目标（候选评分），在灵活性和正确性之间找到了平衡。
