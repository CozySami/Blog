## 6.4 RankMixer: 硬件效率优化 读书笔记

[原文链接](https://datawhalechina.github.io/fun-rec/chapter_6_scaling/4.rankmixer.html)

传统深度学习推荐模型（DLRM）在 GPU 上的 Model FLOPs Utilization（MFU）通常只有 4-5%，而大语言模型可达到 40-60%。这个十倍的效率差距直接导致推荐模型无法享受 scaling law 的红利。

这一问题的根源在于传统 DLRM 继承自 CPU 时代的架构设计：
1. **核心操作以 memory-bound 为主**：embedding lookup、特征交叉等操作的访存量远大于计算量
2. **计算图高度碎片化**：多个独立模块串联，kernel launch 开销累积成显著瓶颈
3. **无法充分利用 Tensor Core**：大部分操作是小规模向量运算或不规则访存

RankMixer 通过 **hardware-aware 的架构设计**解决这一问题，核心原则是**从硬件特性反推架构设计**：用 Token Mixing 替代 Self-Attention 以降低复杂度，用 Per-Token FFN 捕捉特征异质性，用 Sparse MoE 实现参数高效的模型扩展。

### 6.4.1 Token Mixing 机制

Self-Attention 的计算复杂度为 \(O(T^2D)\)，在推荐场景中 token 数量可达数百上千，这个 \(T^2\) 项成为显著瓶颈。RankMixer 的核心洞察是：**推荐任务需要的是 token 之间的信息混合（mixing），而非基于相似度的动态加权（attention）**。

Token Mixing 在特征维度而非 token 维度进行混合：
1. 将每个 token 分解为多个 head
2. 对每个 head，将所有 token 的该 head 部分拼接起来
3. 改变数据组织方式，从"按 token 组织"变为"按 head 组织"

复杂度从 \(O(T^2D + TD^2)\) 降低到 \(O(TD)\)。关于如何保证捕捉 token 之间交互，答案在于**多层堆叠**：单层实现 feature-level mixing，堆叠多层时可以建模 \(L\) 阶多项式交互。

### 6.4.2 Per-Token FFN

标准 Transformer 的 FFN 对所有 token 使用相同权重，这在语言模型中是合理的，但推荐系统的特征完全不同：用户 ID 表示隐式偏好、物品类目是粗粒度分类、点击率服从 long-tail 分布。RankMixer 的核心设计是：**每个 token 有独立的 FFN 参数**。

这种设计与 MMoE 有本质区别：MMoE 中多个 expert 共享同一输入，输出是 expert 的加权组合；Per-Token FFN 中每个 token 有独立的输入和独立的 FFN。

虽然参数量增加了 \(T\) 倍，但**计算复杂度不变**。关键在于增加的参数是"专门化"的，每块参数只服务于特定 token，学习效率更高。

### 6.4.3 Sparse MoE 扩展

在推荐场景中，实现有效的 expert specialization 面临三大挑战：高维稀疏特征向量缺乏清晰的 cluster structure、数据极度不均衡导致 expert overload 和 underutilization、推理时的负载波动。

RankMixer 采用了两种互补的训练策略：

1. **ReLU Routing**：允许每个 token 激活不同数量的 expert，通过正则化项控制稀疏度
2. **Dense-Training / Sparse-Inference（DTSI-MoE）**：训练时使用 dense router 激活所有或大部分 expert，推理时使用 sparse router 只激活少数 expert

### 6.4.4 MFU 提升的关键

RankMixer 实现高 MFU 的关键在于：**所有核心操作都是 compute-bound 的大矩阵乘法**。Token Mixing 和 PFFN 占据约 85% 计算时间，全部是 GEMM 操作，可高效利用 GPU 的 Tensor Core。

相比之下，传统 DLRM 中 Embedding lookup 占 40% 时间（memory-bound），小 kernel 操作占 35% 时间（MFU < 10%），kernel launch overhead 占 20% 时间，有效 GEMM 计算仅占 5%。这解释了为什么传统 DLRM 的 MFU 只有 4-5%，而 RankMixer 可达 45%。

通过 hardware-aware 的架构设计，RankMixer 将推荐模型从碎片化设计转变为统一架构范式，让推荐模型真正成为 GPU 的"第一类公民"，打开了可持续的 scaling 路径。
